*** Git commits
#+NAME: prompts/git-commit
#+begin_src markdown :tangle ~/.ai/prompts/git-commit.md
  When making git commits, you will respect the following commit message format:
  ```
  [<TICKET-ID>:] <TYPE>: <SUBJECT>

  [<BODY>]
  ```

  The <TICKET-ID> is optional and should only be included if there is a related
  ticket in an issue tracker, e.g. Jira. You will determine if this is the case
  by looking at the branch name if available. The <TYPE> should be one of the keywords from the
  semantic commit message specification.
  An additional rule you should follow:
  - If there are user-facing changes, you MUST use either feat or fix
  - In other cases, use your best judgement for which one fits best

  The <SUBJECT> should be a short description of the change, ideally no more than
  50 characters. It should be written in the imperative mood, as if giving a
  command. The subject should describe the intent of the change, not a verbatim description
  of the change itself - that can be glanced from the diff.

  If a change is an intermediate step (i.e. the project is not yet in a
  working/testable state), preferably append [skip ci] to the commit message.

  Use <BODY> sparingly, only when you need to explain the change in more detail
  and the subject does not suffice.

  Examples:
  - Branch: feature/ABC-1234-add-login-endpoint
    Commit message:
    ```
    ABC-1234: feat: Add endpoint /v1/auth/login

    - Authenticate users using JWT token
    - Document in OpenAPI spec

    Payload:
      {
        "username": "string",
        "password": "string"
      }
    ```
  - Unspecified branch
    Commit message:
    ```
    refactor: Correct typo in code comment
    ```
    This is a simple commit, so no body is needed.
  - Branch: bugfix/DE-56-fix-unconfigured-integrations
    Commit message:
    ```
    DE-56: fix: Handle unconfigured integrations gracefully

    - Add mutex to prevent desynchronized access where config could be null
    - Add unit test to cover this case
    ```
  - Branch: docs/update-contributing-guidelines
    Commit message:
    ```
    docs: Update contributing guidelines
    ```
  - Unspecified branch
    Commit message:
    ```
    feat: Add prototype data collector [skip ci]

    - Use Kafka for ingress
    - Store data in-memory
    - Persistence is WIP
    ```
    This is an intermediate step that is not yet ready for testing, so it is marked with `[skip ci]`.

  - Unspecified branch
    Single function added
    Commit message:
    ```
    feat: Add function filterByActivity

    The function filters a list of devices by their last activity timestamp.
    ```
    If the name of the function is short and self-explanatory, prefer using the name
    instead of a natural language description of the function in the subject.

  Miscellaneous other examples
  ```
  feat: Add HARDCODED_REPORTING_INTERVALS_MILLIS with 'D5-00-01'
  ```

  Now write the commit message. DO NOT ADD MARKDOWN CODE BLOCK FENCES AROUND IT.
#+end_src

  Address review feedback on $ARGUMENTS.

  $ARGUMENTS may be a GitHub PR URL/number, or a specific review-comment URL.

  ## Workflow
  1. Fetch the relevant review comments using `gh api`:
     - For a PR: list all unresolved review threads and top-level review bodies.
     - For a single comment URL: fetch that comment plus the rest of its thread.
     Use the GraphQL API to get thread `isResolved` state — REST does not
     expose it.
  2. For each comment, read the referenced file/line and form a brief opinion:
     agree / disagree / needs clarification. Present a numbered list to the
     user with each comment's gist and your take.
  3. Wait for the user to decide which comments to address, dismiss, or push
     back on. Do not start changing code until they respond.
  4. For comments the user wants fixed: make the changes, show the diff, wait
     for approval, then commit using the repo's commit conventions.
  5. Reply to every addressed comment via `gh api`:
     - Accepted: brief reply, reference the commit SHA.
     - Dismissed: brief explanation of why.
     Then resolve the thread via the GraphQL `resolveReviewThread` mutation.

  ## Rules
  - Never resolve a thread without replying first.
  - Never push to the remote unless the user explicitly asks.
  - If a comment is ambiguous, ask the user before deciding agree/disagree.

*** Atomic commit
#+begin_src markdown :tangle ~/.ai/skills/atomic-commit/SKILL.md
  Classify the current git changeset into atomic subsets suitable for individual
  commits and commit them one by one.

  ## Workflow

  1. Run `git diff` (include staged + unstaged) to see all uncommitted changes.
  2. Classify the changes into atomic, independent subsets. Each subset should be
     a logically coherent unit of change (e.g. one feature, one fix, one config
     change per file/topic). Present the list to the user with a short title and
     the files/hunks involved.
  3. For each subset the user selects:
     a. Stage **only** the relevant hunks (use `git add -p` with scripted
        answers, or `git add <file>` when the whole file belongs to the subset).
     b. Show the staged diff with color (`git diff --staged --color=always`) so
        the user can review it.
     c. Wait for the user to approve before committing.
     d. Commit with a well-formed message following the repo's commit conventions.
  4. After all selected subsets are committed, show the remaining unstaged changes
     if any.

  ## Rules
  - NEVER commit without showing the WHOLE staged diff first.
  - Always enable color when printing diffs (`--color=always`).
  - If a file contains hunks belonging to different subsets, use `git add -p`
    with non-interactive scripted input to stage only the relevant hunks.
  - Follow the commit message conventions already established in the repository
    (inspect recent `git log` if unsure).

*** MCP servers
Registers user-scope MCP servers via the =claude mcp= CLI. Idempotent: removes
any existing entry with the same name before re-adding. Avoids editing
=~/.claude.json= directly (it holds session state).
#+begin_src bash :tangle ~/.local/bin/claude-mcp-bootstrap.sh :shebang "#!/usr/bin/env bash"
  set -eu

  register() {
      name=$1; shift
      claude mcp remove --scope user "$name" >/dev/null 2>&1 || true
      claude mcp add --scope user "$name" -- "$@"
  }

  # NEVER ADD AN `npx <pkg>` ENTRY WITHOUT LOOKING UP THE LATEST VERSION
  # (`npm view <pkg> version`) AND PINNING IT EXPLICITLY (`<pkg>@<version>`).
  # UNPINNED npx FETCHES ARE UNTRUSTED AND CAN CHANGE UNDER YOU.

  # excalidraw: stdio MCP server (yctimlin/mcp_excalidraw fork) that syncs to a
  # live browser canvas. The canvas itself runs as a docker service — see
  # background.org -> Excalidraw canvas. View it at
  # http://127.0.0.1:$HARIS_EXCALIDRAW_PORT (port comes from environment.d).
  register excalidraw sh -c 'exec env "EXPRESS_SERVER_URL=http://127.0.0.1:$HARIS_EXCALIDRAW_PORT" ENABLE_CANVAS_SYNC=true npx -y mcp-excalidraw-server@1.0.7'

  # emacs: bridges to the running Emacs's mcp-server via Unix socket (see emacs.org).
  # Wrapped in `systemd-run --user --pipe` so stderr (socat -v traffic dump) lands
  # in the journal. Inspect with `journalctl --user -u 'claude-mcp-emacs-*' -f`.
  register emacs sh -c 'exec systemd-run --user --pipe --quiet --collect --unit="claude-mcp-emacs-$$" -- socat -v - "UNIX-CONNECT:$XDG_RUNTIME_DIR/emacs-mcp-server.sock"'

**** Idle notification (Stop)
#+begin_src bash :tangle ~/.local/lib/haris/claude-hook-notify-idle.sh
  set -eu
  # Prevent recursion when this hook fires for the nested haiku summarizer.
  [ -n "${CLAUDE_HOOK_SUMMARIZER:-}" ] && exit 0

  # Debug log via FIFO. Reader attaches with `cat /tmp/claude-hook-notify-idle.fifo`.
  # We self-open r/w so writes never block whether or not a reader is present.
  FIFO=/tmp/claude-hook-notify-idle.fifo
  [ -p "$FIFO" ] || mkfifo -m 0600 "$FIFO" 2>/dev/null || true
  exec 9<>"$FIFO" 2>/dev/null || true
  log() { printf '[%s] %s\n' "$(date +%H:%M:%S)" "$*" >&9 2>/dev/null || true; }

  session_id=$(jq -r '.session_id')
  digest=$(printf '%s' "$session_id" | sha256sum | cut -c1-6)
  log "start session=$session_id"

  ws_file="${XDG_RUNTIME_DIR:-/tmp}/claude-ws/$session_id"
  ws=""
  [ -r "$ws_file" ] && ws=$(cat "$ws_file")

  # Marker is touched by claude-hook-clear-idle.sh when the user submits a new
  # prompt. We check it (a) to abort the summarizer mid-flight, (b) to suppress
  # the final notify-send so it doesn't re-pop after the user has moved on.
  marker_dir="${XDG_RUNTIME_DIR:-/tmp}/claude-notify"
  mkdir -p "$marker_dir"
  marker="$marker_dir/$session_id.cleared"
  notif_file="$marker_dir/$session_id.notif"
  rm -f "$marker"
  out=$(mktemp -t claude-summary.XXXXXX)
  trap 'rm -f "$marker" "$out"' EXIT

  summary_title="Claude${ws:+ [$ws]} $digest"
  notify() {
      printf '%s\n%s' "$summary_title" "$1" > "$notif_file" 2>/dev/null || true
      notify-send -t 60000 --hint="string:x-dunst-stack-tag:claude-$session_id" \
          "$summary_title" "$1" 2>/dev/null || true
  }

  notify "Done, generating summary..."

  # Run the summarizer in its own pgrp (setsid) so a marker-triggered
  # `kill -- -$pid` reaps python *and* its nested `claude --model haiku` child.
  setsid bash <<OUTER > "$out" 2>/dev/null &
  CLAUDE_HOOK_SUMMARIZER=1 LOG_FIFO="$FIFO" python3 - "$session_id" <<'PY'
  <<claude-hook-summarizer-py>>
  PY
  OUTER
  sum_pid=$!

  while kill -0 "$sum_pid" 2>/dev/null; do
      if [ -e "$marker" ]; then
          kill -TERM -- "-$sum_pid" 2>/dev/null || true
          break
      fi
      sleep 0.2
  done
  wait "$sum_pid" 2>/dev/null || true

  if [ -e "$marker" ]; then
      log "cleared before summary completed"
  else
      title=$(cat "$out")
      log "notify ws=${ws:-<none>} title=${title:-<empty>}"
      notify "${title:-Awaiting your input}"
  fi
#+end_src
***** Summarizer
#+name: claude-hook-summarizer-py
#+begin_src python :tangle no
  import json, glob, os, sys, subprocess, time
  session_id = sys.argv[1]
  FIFO = os.environ.get('LOG_FIFO', '')

  def log(msg):
      if not FIFO:
          return
      try:
          fd = os.open(FIFO, os.O_WRONLY | os.O_NONBLOCK)
          os.write(fd, (time.strftime('[%H:%M:%S] ') + msg + '\n').encode())
          os.close(fd)
      except OSError:
          pass

  f = glob.glob(os.path.expanduser(f'~/.claude/projects/**/{session_id}.jsonl'), recursive=True)
  if not f:
      log('no jsonl found')
      sys.exit()
  log(f'jsonl={f[0]}')

  def extract_text(msg):
      content = msg.get('content', '')
      if isinstance(content, str):
          return content
      return ' '.join(c.get('text', '') for c in content
                      if isinstance(c, dict) and c.get('type') == 'text')

  last_user = ''
  last_asst = ''
  with open(f[0]) as fh:
      for line in fh:
          try:
              obj = json.loads(line)
          except Exception:
              continue
          t = obj.get('type'); msg = obj.get('message', {})
          text = extract_text(msg).strip()
          if not text:
              continue
          if t == 'user':
              if text.startswith('<') or text.startswith('[Request interrupted'):
                  continue
              last_user = text
          elif t == 'assistant':
              last_asst = text

  log(f'last_user[{len(last_user)}]={last_user[:60]!r}')
  log(f'last_asst[{len(last_asst)}]={last_asst[:60]!r}')

  fallback = last_user[:80]
  if not last_user and not last_asst:
      log('empty exchange, using fallback')
      print(fallback); sys.exit()

  excerpt = f"USER:\n{last_user[:2000]}\n\nASSISTANT:\n{last_asst[:2000]}"
  prompt = (
      "Below is the latest exchange from a Claude Code session that just went idle, "
      "awaiting the user's next input. Write a 5-10 word memory-aid summary that lets "
      "the user instantly recall what THIS session is about when they glance at a "
      "desktop notification. Focus on concrete nouns (specific file, task, decision, "
      "blocker) - never generic verbs like 'discussing', 'helping', 'working on'. "
      "Output ONLY the summary line. No quotes, no preamble, no trailing period.\n\n"
      + excerpt
  )
  t0 = time.time()
  try:
      r = subprocess.run(
          ['claude', '--model', 'haiku', '-p', prompt],
          capture_output=True, text=True, timeout=30,
      )
      dt = time.time() - t0
      log(f'haiku rc={r.returncode} dt={dt:.1f}s stdout={r.stdout.strip()[:120]!r} stderr={r.stderr.strip()[:120]!r}')
      out = (r.stdout or '').strip().splitlines()
      line = (out[0] if out else '').strip().strip('"').strip("'")
      final = line[:80] if line else fallback
      log(f'final title={final!r}')
      print(final)
  except Exception as e:
      log(f'haiku exception={e!r}')
      print(fallback)
#+end_src
**** Clear idle notification (UserPromptSubmit)
Dismisses the Stop hook's lingering idle popup when the user submits a new
prompt.
#+begin_src bash :tangle ~/.local/lib/haris/claude-hook-clear-idle.sh
  set -eu
  session_id=$(jq -r '.session_id')
  # Marker tells a still-running notify-idle to abort its summarizer and skip
  # the final notify-send. notify-idle owns cleanup via its EXIT trap.
  marker_dir="${XDG_RUNTIME_DIR:-/tmp}/claude-notify"
  mkdir -p "$marker_dir"
  : > "$marker_dir/$session_id.cleared"
  # Replace the stacked idle popup with a 1ms-timeout notification so dunst
  # dismisses it immediately. Same x-dunst-stack-tag as in notify-idle.
  # Reuse the last title/body written by notify-idle so dunst's history shows
  # the real summary rather than an empty entry.
  notif_file="$marker_dir/$session_id.notif"
  title=""; body=""
  if [ -r "$notif_file" ]; then
      title=$(sed -n '1p' "$notif_file")
      body=$(sed -n '2,$p' "$notif_file")
  fi
  notify-send -t 1 --hint="string:x-dunst-stack-tag:claude-$session_id" \
      "$title" "$body" 2>/dev/null || true

**** Bash command tips (PreToolUse)
Pattern-matches the bash command Claude is about to run and, on match, blocks
it with =permissionDecision: deny= and the tip as the reason — Claude sees the
message and reformulates before the user permission prompt fires. Mirrors the
=claude-tip= hints embedded in shell wrappers (see =scripts.org=), but
intercepts pre-execution so Claude can act on the hint.
#+begin_src python :tangle ~/.local/lib/haris/claude-hook-bash-tips.py :shebang "#!/usr/bin/env python3"
  import argparse, json, os, re, shlex, subprocess, sys

  data = json.load(sys.stdin)
  cmd = data.get("tool_input", {}).get("command", "")
  cwd = data.get("cwd") or os.getcwd()


  def deny(reason):
      print(reason, file=sys.stderr)
      sys.exit(2)


  def tip_git_dash_C():
      """`git -C <dir>` where <dir> is the CWD or an ancestor -> -C is redundant."""
      try:
          tokens = shlex.split(cmd)
      except ValueError:
          return
      if "git" not in tokens:
          return
      argv = tokens[tokens.index("git") + 1 :]
      p = argparse.ArgumentParser(add_help=False)
      p.add_argument("-C", dest="dir", action="append")
      args, _ = p.parse_known_args(argv)
      for target in args.dir or []:
          real = os.path.realpath(target)
          if (cwd + "/").startswith(real + "/"):
              deny(f"you only need `-C` to act on a git worktree that's not your CWD. "
                   f"Re-run without `-C {target}`.")


  def tip_format_long_command():
      """Dense one-liners are unreadable in the transcript. If shfmt expands the
      command into more lines than the original, tell Claude to re-run the
      formatted version. Gated on length>120 OR >=2 chain operators (&&, ||, ;, |)
      to skip the shfmt fork for trivially short commands."""
      long_enough = len(cmd) > 120
      chain_count = len(re.findall(r'&&|\|\||;|\|', cmd))
      has_chains = chain_count >= 2
      if not (long_enough or has_chains):
          return
      try:
          r = subprocess.run(
              ['shfmt', '-i', '2'], input=cmd,
              capture_output=True, text=True, timeout=2,
          )
      except (FileNotFoundError, subprocess.TimeoutExpired):
          return
      if r.returncode != 0:
          return
      formatted = r.stdout.rstrip('\n')
      orig_lines = cmd.count('\n') + 1
      new_lines = formatted.count('\n') + 1
      if new_lines > orig_lines:
          deny("this command is dense and unreadable in the transcript. "
